# 百万级法眼AI：架构演进指南

> 本文档回答：从当前1.2万条案例扩展到百万级，法眼AI需要做哪些升级

---

## 一、核心认知：量变引发质变的节点

从1万到100万，不是简单的线性扩展。有几个关键的**量变引发质变**节点：

- **JSON文件不可行**：100万条案例JSON文件约3-4GB，内存无法全量加载
- **单进程BM25瓶颈**：百万级语料全量扫描，延迟从毫秒级上升到秒级
- **口语 vs 法言法语gap扩大**：小规模数据靠BM25的精确匹配可以cover，百万级需要向量语义检索补充
- **数据质量参差不齐**：手工检查不可能，必须建立自动化流水线

---

## 二、当前状态 vs 百万级对比

| 维度 | 当前（1.2万） | 百万级 |
|------|------------|-------|
| 存储 | JSON文件，内存加载 | PostgreSQL分表 |
| 检索 | 单进程BM25/TF-IDF | PostgreSQL粗排 + ES/向量精排 |
| RAG | 全量内存BM25扫描 | 双路混合检索 + RRF融合 |
| 数据管道 | 脚本手动处理，一次性 | 增量清洗流水线 |
| 法规库 | 无 | 独立索引，可选接入 |
| 评估 | 无 | 基础评测集（可选） |
| 基础设施 | 单进程Python | 微服务化（可单机） |
| 分布式 | 不需要 | 不需要（单机可支撑） |

**百万级核心原则：能用单机的绝不用集群。**

## 三、四大核心升级

### 3.1 存储：从JSON到PostgreSQL

当前所有案例全量加载到内存，100万条JSON不可行。

```sql
-- 建议表设计
CREATE TABLE cases (
    id              TEXT PRIMARY KEY,
    case_number     TEXT NOT NULL,
    title           TEXT NOT NULL,
    court           TEXT,
    case_date       DATE,
    case_type       TEXT,        -- 刑事/民事
    cause_of_action TEXT,        -- 案由，高频过滤字段
    basic_facts     TEXT,
    ruling_points   TEXT,
    related_laws    TEXT[],
    full_text       TEXT,
    region          TEXT,        -- 从court提取省份，用于地域加权
    metadata        JSONB,       -- 审级、案由类型等标签
    embedding       vector(1536), -- 可选，向量直接存PG
    created_at      TIMESTAMP DEFAULT NOW()
);

-- 关键索引
CREATE INDEX idx_cause ON cases(cause_of_action);          -- 案由精确过滤
CREATE INDEX idx_court ON cases(court);                     -- 法院/地域
CREATE INDEX idx_date  ON cases(case_date DESC);            -- 时间排序
CREATE INDEX idx_region ON cases(region);                   -- 地域加权
CREATE INDEX idx_fulltext ON cases USING GIN (
    to_tsvector('zhparser', basic_facts || ' ' || ruling_points)
);
```

**关键设计理由：**
- `cause_of_action`（案由）是最高频的过滤条件，**必须建B-tree索引**
- `region`（省份）从court字段提取，用于地域加权过滤
- `embedding`向量可以直接存PostgreSQL的`pgvector`扩展，不需要独立的向量库
- full_text不建全文索引（太大），只对basic_facts和ruling_points建GIN索引

**百万级PostgreSQL性能参考：**
- 8核CPU + 32GB RAM + SSD，可平稳支撑百万级案例的存储和检索
- 查询延迟：PG过滤 < 50ms，向量检索 < 100ms，总延迟 < 200ms

### 3.2 检索：双层架构（粗排 → 精排）

```
用户案情描述
    │
    ▼
意图识别（规则 + 简单LLM）
    │
    ├─ 提取案由（cause_of_action）
    ├─ 提取法院所在省（region）
    ├─ 提取时间范围（case_date）
    └─ 提取关键事实词（用于ES检索）
    │
    ▼
第一层：PostgreSQL 粗排（毫秒级）
    │
    WHERE cause_of_action = '民间借贷纠纷'          -- 案由精确过滤
      AND region = '四川省'                          -- 地域过滤（可选）
      AND case_date >= '2020-01-01'                 -- 时间过滤（可选）
    ORDER BY case_date DESC
    LIMIT 500                                       -- 候选集控制在500条
    │
    ▼
第二层：ES-BM25 / 向量精排
    │
    对500条候选做 BM25 + 字段加权排序
    取 top 5-10
    │
    ▼
输出：Top-5 最相关案例 + 关联法条（可选）
```

**为什么不是直接ES检索？**

ES单字段全文检索百万级完全没问题，但多条件组合过滤（案由 + 地域 + 时间 + 内容相关性排序）在ES里实现复杂且资源消耗高。

用PG做精确过滤 + ES/向量做语义排序的组合，性价比最高：
- PG的B-tree索引对等值过滤极其高效
- 过滤后候选集从100万缩小到500条
- 500条再做ES/向量精排，延迟可控

### 3.3 RAG：双路混合检索

```
用户查询："借款不还，对方失踪了，还能起诉吗"
    │
    ▼
查询改写（Query Decomposition）
    │
    分解为：
    - 案由：民间借贷纠纷
    - 关键问题：被告下落不明 / 起诉条件
    │
    ▼
┌──────────────────────────────────────┐
│           双路检索（并行）              │
├──────────────────┬───────────────────┤
│  路1: PostgreSQL  │  路2: 向量检索     │
│  关键词检索       │  （pgvector/FAISS）│
│                  │                    │
│  jieba分词        │  embedding模型     │
│  字段加权         │  text-embedding-3 │
│  BM25排序         │  small / 法律微调版│
│  Top-20召回       │  Top-20召回        │
└──────────────────┴───────────────────┘
    │
    ▼
RRF融合（Reciprocal Rank Fusion）
    │
    score = 0.6 × rank_bm25 + 0.4 × rank_vector
    │
    ▼
MMR多样性重排
    │
    ▼
Top-5 输出
```

**为什么百万级必须引入向量检索？**

BM25擅长精确术语匹配（"盗窃罪"、"民间借贷"），但用户输入往往是口语化描述。

百万级语料中，口语表达和法言法语的语义gap比小规模数据更明显：
- 用户说："老公出轨了能离婚吗"
- BM25只能匹配含"离婚"关键词的文书
- 向量检索能匹配"婚姻关系破裂"、"配偶不忠"等语义相关但表述不同的文书

**RRF融合公式：**
```python
def rrf_fusion(bm25_rank, vector_rank, k=60):
    return 0.6 * (1 / (k + bm25_rank)) + 0.4 * (1 / (k + vector_rank))
```
k=60是经验值，避免排名差距过大时低排名结果被完全压制。

### 3.4 数据管道：增量清洗流水线

百万级数据不可能每次全量处理，必须建立增量流水线。

```
新增裁判文书（每日/每周增量）
    │
    ▼
1. 提取脚本 → 结构化JSON
    │  从原始文书提取：案号、法院、案由、当事人、事实、判决结果
    │
    ▼
2. NER脱敏（ spaCy / 法律微调NER模型）
    │  识别：人名、地名、身份证号、银行卡号 → 替换为"某甲"/"某地"
    │  多被告消歧：同一句中的多个"某甲"需要正确关联到不同当事人
    │
    ▼
3. 质量过滤
    │  - basic_facts 长度 < 50字 → 丢弃（程序性文书）
    │  - ruling_points 为空 → 标记"需审核"（不丢弃，但不入库）
    │  - 乱码率 > 5% → 丢弃
    │  - 重复检测：SimHash 快速去重
    │
    ▼
4. 入库 PostgreSQL
    │
    ▼
5. 向量索引更新（FAISS IVFFlat，每日批量）
    │
    ▼
6. 法规关联（可选）
       通过 related_laws 字段建立关联
       提前标注法条时效（现行/已废止）
```

**关键组件选型：**

| 组件 | 选型 | 理由 |
|------|------|------|
| 脱敏 | 法律微调NER + 规则后处理 | 正则替换会破坏多被告上下文 |
| 全文检索 | PostgreSQL GIN (zhparser) | 不需要额外的ES集群 |
| 向量检索 | FAISS IVFFlat 或 pgvector | 单机可支撑百万向量 |
| 定时任务 | cron + Python脚本 | 简单可靠，不需要复杂调度系统 |

## 四、与千万级的关键区别

| 维度 | 百万级 | 千万级 |
|------|-------|-------|
| 向量检索 | FAISS（单机可支持100万向量） | Milvus（分布式） |
| 模型训练 | 可以不做，继续用MiniMax API调用 | 需要三阶段训练（CPT+SFT+DPO） |
| 法规库 | 静态库，定时同步即可 | 实时更新 + 时间戳过滤（已废止法条排除） |
| 基础设施 | 单机PG + FAISS可支撑 | 需要ES集群 + 向量集群 + GPU训练集群 |
| 数据管道 | 增量批处理即可 | 需要流式处理 + 分布式存储 |
| 评估体系 | 可选，有则更好 | 必须（类案Hit Rate benchmark） |
| 规则引擎 | Python类 + 独立服务均可 | 必须独立服务（版本化+热更新） |

**千万级的本质区别：单机的极限已经不够用了。**

- 100万向量：FAISS IVFFlat 单机秒级检索
- 1000万向量：需要Milvus分布式分片 + 向量索引分布式构建
- 千万级案例训练：需要多卡GPU集群（8×A100 80G）+ 分布式训练框架

## 六、落地建议

**阶段一：PostgreSQL化（1-2周）**
1. 设计并创建 `cases` 表，建好索引
2. 将现有JSON数据导入PG
3. 实现基于PG的条件过滤查询
4. 验证性能（目标：查询 < 200ms）

**阶段二：双路检索（1-2周）**
1. 部署FAISS，生成百万级案例向量
2. 实现PG过滤 + FAISS精排的双层架构
3. 调优RRF融合权重
4. 实现MMR多样性重排

**阶段三：数据管道（2-4周）**
1. NER脱敏脚本
2. 增量入库流程
3. FAISS索引增量更新
4. 质量监控与报警

**阶段四（可选）：模型微调**
1. 如果检索+生成效果已达瓶颈，再考虑法律基座持续预训练
2. 建议从单一案由（如民间借贷）入手验证

---

## 七、一句话总结

> 百万级法眼AI，**存储用PostgreSQL + 检索用PG过滤+FAISS向量双路 + 数据管道增量跑**。不需要训练基座模型，不需要分布式集群，用合理的架构设计把单机性能压榨到极致。
