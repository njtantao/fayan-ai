# 法律大模型千万级案例训练：法眼AI实现指南

> 本文档基于通用法律AI训练方法论与法眼AI项目代码/文档综合整理

---

## 一、核心认知

用"千万条案例"训练法律大模型，是目前法律科技领域的核心赛道，也是最具挑战性的工程之一。千万级的案例（通常指裁判文书网等公开的司法数据）蕴含着极高的商业和学术价值，但要把这些非结构化的文本转化为具备法律推理能力的"数字法官"或"超级律师"，绝不仅仅是"把数据喂给模型"那么简单。

法律大模型有句行话：**"模型的天花板在预训练，预训练的天花板在数据清洗"**。千万条案例约几百GB到1TB的纯文本，质量参差不齐。

---

## 二、数据工程：决定模型上限的"炼丹"第一步

### 2.1 脱敏与合规（生死线）

- **个人信息脱敏**：必须剔除或替换当事人姓名、身份证号、住址、银行账号等。简单的正则替换会破坏上下文（如"张三"替换为"某甲"后，多被告案件会变成"某甲与某乙"），需要用命名实体识别（NER）技术做智能脱敏。
- **合规审查**：确保数据来源合法，符合《数据安全法》和《个人信息保护法》。

### 2.2 深度解析与结构化（苦力活）

裁判文书是半结构化文本，需要精准提取：
- 案件事实
- 诉辩主张
- 法院观点
- 裁判结果
- 引用法条

难点在于"本院认为"部分往往与案件事实交织，需要通过NLP或规则引擎将其剥离，形成"事实-推理-结论"的三段式对齐数据。

### 2.3 数据去重与降噪

- **去重**：千万级数据中存在大量同案不同号的文书、串案文书。需使用MinHash、SimHash等算法去重，防止模型记忆重复模式。
- **去噪**：剔除程序性文书（如撤诉裁定、管辖权异议裁定），保留有实体审理内容的文书；剔除文书上网产生的乱码、HTML标签。

## 三、模型训练策略：三阶段渐进法

直接拿千万案例从零训练一个基座模型成本极高且不必要。目前的主流做法是**"通用基座 + 法律领域持续预训练 + 案例指令微调"**。

### 第一阶段：法律领域持续预训练

- **目标**：让通用大模型（如Qwen, Baichuan, Llama-3）"懂法律黑话"，建立法律知识底座
- **数据配比**：千万级案例 + 法律法规 + 法学教材 + 法律问答。不要只喂案例，否则模型会只懂法官视角，不懂律师和当事人视角
- **训练方式**：下一词预测（CPT）。学习率应设低（通常为基座预训练的1/10左右），避免"灾难性遗忘"

### 第二阶段：基于案例的指令微调

- **目标**：让模型学会"按指令办事"，如"撰写起诉状"、"提取争议焦点"、"预测判决"
- **数据构造**（核心资产）：
  - 示例1（争议焦点提取）：`Input=案件事实；Output=本案争议焦点为：1... 2...`
  - 示例2（类案裁判预测）：`Input=原告主张...被告抗辩...；Output=法院可能支持...理由是...`
- **关键点**：千万不能让模型去预测赔偿的具体精确数字（如25314.5元），大模型本质是概率模型，做精确数值回归极易产生幻觉。应引导其输出计算逻辑和法理依据

### 第三阶段：对齐与价值观注入（RLHF/DPO）

- **目标**：确保模型的输出符合社会主义法治理念，不输出反常识、反法律的结论
- **做法**：构建法律偏好数据集。例如，对于同一个法律咨询，答案A引用了现行有效法律且逻辑严密，答案B引用了已废止法律或胡编乱造法条，通过DPO让模型学会偏好A

## 四、千万级案例训练的"致命坑"与对策

### 4.1 幻觉问题：伪造法条与编造案号

- **现象**：模型一本正经地胡说八道，编造《民法典第9999条》或不存在的指导性案例
- **对策**：引入RAG（检索增强生成）架构。大模型只负责推理，法条和案例的召回交给向量数据库（如Milvus）。让模型"开卷考试"，基于检索到的真实法条生成答案

### 4.2 时效性问题：法律更迭

- **现象**：用2021年前的案例训练，模型在2024年还在适用《合同法》（已废止，现适用《民法典》）
- **对策**：在数据清洗时，对旧法案例进行标注或降权；在RAG层做时间戳过滤，优先召回适用新法的案例

### 4.3 "同案不同判"的冲突

- **现象**：千万案例中，相似事实可能有截然不同的判决（各地裁判尺度不一）
- **对策**：在指令微调时，不要给模型唯一的标准答案，而是提供多视角的裁判观点。让模型学会输出"实践中存在两种观点：一种认为...另一种认为...目前主流观点倾向于..."

### 4.4 长上下文处理

- **现象**：复杂案件的卷宗动辄数万字，超出模型窗口限制
- **对策**：采用支持长文本的基座模型（如Qwen-1M, GLM-128K），或使用Map-Reduce策略，先让小模型分块提取案件事实，再让大模型基于提取的事实做推理

## 五、评估体系：如何证明模型真的"懂法"

不要只看模型生成的文本是否流畅，必须建立客观的法律能力评测集：

| 评测方式 | 内容 |
|---------|------|
| **客观题评测** | 使用法考（国家统一法律职业资格考试）历年真题、法官检察官入额考试题，测试法条记忆与基础逻辑 |
| **信息抽取评测** | 随机抽取1000篇未见过的文书，测试模型提取"诉讼请求、抗辩理由、裁判结果"的F1分数 |
| **类案检索与匹配评测** | 给定案件事实，让模型从千万案例中检索出Top-10相似案例，计算与专业法官标注的Hit Rate和MRR |
| **人类评估** | 邀请执业律师盲测，对比模型输出与初级律师的工作成果 |

## 六、法眼AI当前架构 vs 千万级目标架构

### 6.1 当前架构（1.2万条案例）

```
用户输入案情
    ↓
1. 分类（规则关键词优先，LLM辅助）
2. 复杂度判定（规则引擎）
3. 律师介入检查
4. RAG检索（单进程BM25 + TF-IDF + MMR重排）
5. LLM生成（MiniMax API，受限Prompt）
6. 规则校验（禁用词 + 引用有效性）
```

### 6.2 目标架构（千万级案例）

```
用户输入案情
    ↓
应用服务层（微服务）
    ├─ 案情结构化服务
    ├─ 复杂度判定服务（独立规则引擎）
    ├─ 律师介入服务
    ├─ 类案检索服务
    └─ 文书草稿服务

AI模型层
    ├─ 法律基座模型（三阶段训练）
    └─ 受限生成（Prompt约束 + 语义校验模型）

RAG检索子系统（分布式双路）
    ├─ 向量检索（Milvus）─ 语义相似度
    ├─ 关键词检索（Elasticsearch-BM25）─ 精确术语
    └─ 法规库检索（新增）

规则引擎层
    ├─ 引用校验规则集
    ├─ 禁用词规则集
    ├─ 复杂度判定规则集
    └─ 律师介入触发规则集

知识数据层
    ├─ 法规库（statutes）
    ├─ 类案库（cases）
    ├─ 规则库（rules）
    └─ 用户材料库（uploads）
```

## 七、四大核心升级详解

### 7.1 数据工程：建立自动化清洗流水线

```
原始文书（裁判文书网/最高法数据）
    ↓
1. 脱敏（NER智能实体识别）
    - 当事人姓名、身份证、地址 → 替换为"某甲/某乙"
    - 多被告场景需要消歧
    ↓
2. 结构化提取
    - 案号、法院、案由、判决日期
    - 案件事实、诉辩主张、法院观点、裁判结果
    - 引用法条识别（正则+知识图谱）
    ↓
3. 去重
    - SimHash/MinHash 去除同案不同号
    ↓
4. 质量评分
    - 程序性文书 → 降权或剔除
    - 打标签：审级、地域、年份、案由类型
    ↓
5. 案例库（千万级）+ 法规库
```

### 7.2 检索系统：分布式双路RAG

```
用户案情描述
    ↓
┌─────────────────────────────────┐
│      双路检索（并行）              │
├─────────────────┬───────────────┤
│  语义向量路       │  关键词检索路   │
│  （Milvus/FAISS） │  （ES-BM25）  │
│                 │               │
│  embedding模型  │  jieba分词    │
│  法律语料微调    │  字段加权      │
│  Top-100召回    │  Top-100召回  │
└─────────────────┴───────────────┘
    ↓ RRF融合（Reciprocal Rank Fusion）
    ↓ Top-20
    ↓ MMR多样性重排
    ↓ Top-5输出
```

### 7.3 模型训练：三阶段

```
阶段1: 法律领域持续预训练（CPT）
    基座：Qwen-1.8B / Baichuan-7B / Llama-3-8B
    数据：千万案例 + 法律法规 + 法学教材 + 法律问答
    方法：下一词预测，低学习率
    目标：建立法律知识底座

阶段2: 指令微调（SFT）
    数据：百万级 <Instruction, Input, Output> 对
    关键：不做精确数值回归，输出计算逻辑和法理依据

阶段3: 对齐（DPO）
    数据：偏好数据集（答案A vs 答案B）
    目标：模型学会偏好引用有效法条、逻辑严密的答案
```

### 7.4 规则引擎：独立服务化

```
独立规则引擎服务（Drools / JsonLogic）
    │
    ├─ 引用校验规则集
    │   无效引用 → 删除该条结论
    │   全部失效 → "依据不足"兜底
    │
    ├─ 禁用词规则集
    │   明文列表 + 语义校验模型
    │
    ├─ 复杂度判定规则集
    │   low / medium / high / ultra
    │
    └─ 律师介入触发规则集
        high/ultra → 固定引导语
```

## 八、推荐落地架构

```
[用户输入: 案件描述]
      ↓
[意图识别与改写] → 提取关键词、争议焦点
      ↓
[双路检索 RAG]
  ├─ 法规库检索 (向量库 + 知识图谱: 确保法条准确、现行有效)
  └─ 案例库检索 (千万级裁判文书向量库: 召回相似指导性案例/类案)
      ↓
[法律大模型 (CPT + SFT 炼成的核心)]
  ├─ 输入: 用户描述 + 召回法条 + 召回类案
  └─ 推理: 依据现行法律，参考类案裁判规则，进行逻辑推演
      ↓
[结果校验与输出]
  ├─ 引用溯源: 标注引用的法条和案例来源
  └─ 生成: 法律意见书/裁判文书/起诉状草案
```

**关键比例**：大模型30% + 外挂系统（RAG等）70%

## 九、核心原则

> 法律大模型的核心资产不是"千万条案例"，而是**高质量的清洗与结构化 + 指令微调数据 + 精准的RAG检索层**。案例量是燃料，清洗和检索才是发动机。

> 建议从**特定案由（如民间借贷、交通事故）**开始跑通全流程，验证效果后，再扩展到千万级全量案例。

## 十、与法眼AI现状的关键差异总结

| 维度 | 当前法眼AI | 千万级目标 |
|------|----------|----------|
| 数据规模 | ~1.2万条 | 千万级 |
| 存储 | JSON文件 + 内存 | PostgreSQL + ES + 向量库 |
| 检索 | 单进程BM25/TF-IDF | 分布式双路RAG |
| 模型 | 直接调用MiniMax API | 三阶段训练法律基座 |
| 数据清洗 | 提取脚本手工处理 | 自动化清洗流水线 |
| RAG | 仅案例库 | 案例库 + 法规库 |
| 时效性处理 | 未实现 | 时间戳过滤 + 法规版本管理 |
| 评估体系 | 无 | 完整benchmark |
| 规则引擎 | Python类 | 独立服务（Drools/JsonLogic） |
| 审计机制 | 基础Trace ID | 全链路追踪 + 版本快照 |
