# 法眼AI 系统架构设想

## 一、架构设计总览

法眼 AI 的系统架构遵循**"AI 负责理解与表达、规则负责判断与约束"**的核心设计哲学，整体分为五个主要层次，从前端到后端依次为：用户交互层、应用服务层、AI 模型层、规则引擎层和知识数据层。在各层之间嵌入**校验与审计总线**，确保所有输出可追溯、可拦截、可回滚。

## 二、分层设计与核心职责

### 2.1 用户交互层

**定位**：唯一面向当事人的入口，不设律师端界面（律师端作为后台可选模块）。

**当事人端 Web/App** 包含以下核心页面模块：

| 模块 | 功能描述 | 特殊约束 |
|------|---------|---------|
| 案情输入 | 文字描述/上传文书/语音转文字 | 仅结构化输入，不做开放式问答 |
| 案件分析报告 | 结构化呈现分析结论 | 全程带 citations，每页底部附免责声明 |
| 风险体检报告 | 多维度风险提示 | 不用"会/一定/建议"，用"可能/存在/需关注" |
| 类案参考 | 相似案例摘要与法规关联 | 不输出"你会赢/输"，只输出"此类案法院通常认定…" |
| 文书草稿 | 起诉状/答辩状结构化草稿 | 每段带【需人工确认】标签 |
| 庭审认知 | 流程说明+提问类型 | 不输出应答话术 |

**管理后台** 供内部运营人员维护法规库、案例库、规则库，审核系统输出，抽检审计日志。

**律师引导落地页**：当系统触发"建议律师介入"时，跳转至固定模板页面，不做具体律师推荐，不做疗效承诺，输出固定引导语。

---

### 2.2 应用服务层

应用服务层以**微服务**形式组织，核心服务包括：

**案情结构化引擎**：接收用户输入的自然语言案情，通过 NLP 实体识别（当事人、诉求、事实、证据、时间线）输出结构化 JSON，再经规则引擎校验字段完整性后存入案件数据包。

**类案检索服务**：以结构化案情数据包为查询条件，在类案库中进行双路召回（语义向量 + 关键词），返回相似案例的裁判要点摘要。召回结果须附带案由匹配度、审级权重、地域权重，不返回原始判决全文。

**文书草稿服务**：基于结构化案情数据，填充预设文书模板，输出带【需人工确认】标记的草稿文本。生成过程不带入自由生成内容，仅做模板槽位填充。

**庭审认知服务**：基于案由类型，从规则库中提取庭审流程模板，输出阶段性流程说明和常见提问类型分类，不输出具体应答内容。

**复杂度评估服务**：独立服务，接收案件结构化数据，由规则引擎给出复杂度评级（低/中/高/超高），不经过 AI 模型判断。

---

### 2.3 AI 模型层

AI 模型层是系统中**唯一涉及大模型**的部分，但受到严格的上下文约束和输出约束。

**上下文控制协议**：每次模型调用，上下文仅包含以下内容：
- 用户上传的原始材料（经去标识化处理后）
- RAG 检索召回的法律依据片段（法条/案例摘要/规则）
- 预设的输出模板和结构化 schema
- **严格禁止**带入：模型内置知识、互联网检索结果、摘要+自由知识混用

**RAG 检索子系统**：采用双路召回架构，向量检索（embedding 模型基于法律语料微调）保证语义相关性，关键词检索（BM25）保证法律术语精确性。RAG 的检索范围**严格限定**在自有知识库内，不接入任何外部搜索 API。

**受限生成策略**：Prompt 设计遵循以下原则：
- 明确角色："你是一个法律分析助手，你的所有结论必须附带引用"
- 明确禁止："不要使用模型内置知识，不要编造法条编号"
- 明确边界："如果无法找到依据，明确说明'依据不足'"
- 输出格式：强制 JSON Schema，每条结论带 citations 字段

**语义校验模型**（轻量级）：在 AI 生成结论后、规则引擎校验前，用一个小型分类模型检测输出是否隐含了判断性结论（如"会赢"、"法院会支持"、"应当赔偿"），即使这些词不出现在禁用词表中。这是针对"弱化词包裹强判断"这一隐蔽风险的专门设计。

---

### 2.4 规则引擎层

规则引擎是整个系统的**控制中枢**，承担所有关键判断职责，模型不得凌驾于规则引擎之上。

**统一规则引擎选型**：建议使用 Drools（Java）或基于 JsonLogic 的轻量级规则引擎，便于法律专家参与规则编写和审计。规则文件版本化，与代码分离，支持热更新。

规则引擎管理的核心规则集：

**引用校验规则集**：每条输出的 citations 必须全部在知识库中存在且可解析，任何一条引用不存在则对应结论被删除，不做软通融。

**禁用词规则集**：明文禁用词列表（"胜诉率"、"一定"、"法院会"、"应当（无明确条文）"、"建议你"等）配合语义校验模型输出的隐含判断风险评分，两项任一触发则该条结论被删除。

**复杂度判定规则集**：基于硬编码规则判定，不依赖模型评分：
- 涉案金额阈值（设定绝对值）
- 当事人数量的硬上限
- 证据缺口数量与类型（区分致命缺口与可补强缺口）
- 程序节点数量与审级预期
- 案由是否为复合案由
- 复杂度等级：低 / 中 / 高 / 超高（超出辅助范围）

**律师介入触发规则集**：当复杂度等级为"高"或"超高"，或检测到致命证据缺口，或涉及刑事交叉时，触发律师介入提示，输出固定引导语，不做个性化律师推荐。

**降级兜底规则**：当引用校验删除所有结论后，系统输出固定文案："依据现有材料与知识库，无法形成具有依据的分析结论。建议您咨询专业律师获取进一步指导。"

---

### 2.5 知识数据层

**法规库（statutes）**：存储《民法典》、各诉讼法、司法解释等主干法规。每条法条具备唯一 statute_id、版本号、时效标注。数据来自官方权威来源，人工维护更新，不允许模型补写或延伸解释。

**类案库（cases）**：存储经过结构化处理的裁判文书（去除敏感信息后）。每条案例具备唯一 case_id，结构化字段包括：案由、争议焦点、关键事实、裁判要点摘要、判决结果类型。数据质量控制是关键，建议建立标注团队并设置质量分级。

**规则库（rules）**：存储人工编写的法律逻辑规则，每条具备唯一 rule_id 和可解释的规则描述。规则库是规则引擎的燃料，由法律专家维护，持续扩充。

**向量嵌入库**：为法规、类案、规则生成语义向量，存储于 Milvus 或 FAISS，供 RAG 检索子系统调用。

**数据库选型**：PostgreSQL（结构化数据）+ Elasticsearch（全文检索）+ 向量库（语义检索）。所有数据物理隔离，不与外部网络打通。

## 三、核心处理流程

### 3.1 案情分析主流程

```
用户输入案情
     │
     ▼
案情结构化（NLP实体识别）───▶ 结构化案件数据包
                                      │
                             复杂度判定（规则引擎）───▶ 复杂度等级
                                      │
     ┌───────────────────────────────┼───────────────────────────────┐
     │                               │                               │
     ▼                               ▼                               ▼
  复杂度低/中                    复杂度高                        复杂度超高
     │                               │                               │
     ▼                               ▼                               ▼
  继续分析流程              触发律师引导提示              直接输出律师引导
     │                               │                         + 终止分析
     ▼                                                            │
RAG双路召回（法规/案例/规则）                                            │
     │                                                            │
     ▼                                                            │
AI受限生成（带引用约束）                                              │
     │                                                            │
     ▼                                                            │
语义校验模型（隐含判断检测）                                          │
     │                                                            │
     ▼                                                            │
引用校验器（逐条检查 citations 存在性）                                │
     │                                                            │
     ├──── 引用全部有效 ────▶ 禁用词扫描 ────▶ 结构化JSON输出        │
     │                                                            │
     └──── 存在无效引用 ──▶ 删除该条结论 ──▶ 重新评估 ──┘
                                                              │
                                                    结论全部被删除
                                                              │
                                                              ▼
                                                    输出"依据不足"提示
```

### 3.2 文书草稿子流程

```
输入：结构化案件数据包
     │
     ▼
模板匹配（基于案由）───▶ 选择文书模板（起诉状/答辩状）
     │
     ▼
结构化字段填充（仅模板槽位，不做自由生成）
     │
     ▼
AI辅助补全（仅用于描述性文本，且强制附带引用）
     │
     ▼
【需人工确认】标记注入（所有关键段落）
     │
     ▼
文首免责声明注入
     │
     ▼
输出：带标签的文书草稿（不可直接提交）
```

### 3.3 律师引导触发流程

```
复杂度判定规则集 ──▶ 触发律师介入
     │
     ▼
检查触发原因（可多选）：
  [ ] 涉案金额超阈值
  [ ] 证据存在致命缺口
  [ ] 涉及多方主体
  [ ] 程序节点复杂
  [ ] 涉及刑民交叉
     │
     ▼
生成固定引导语（模板化，不做个性化推荐）
     │
     ▼
输出示例：
"基于目前风险评估，该案件已超出系统智能辅助范围，建议您考虑专业律师支持以降低不确定性。如需了解公共法律服务资源，可通过12348法律服务热线获取帮助。"
```

## 四、安全与审计机制

### 4.1 全链路追踪

每一次用户请求生成唯一 Trace ID，贯穿以下所有环节的日志记录：

| 环节 | 记录内容 |
|------|---------|
| 输入阶段 | 用户材料 hash、输入时间、session ID |
| 结构化阶段 | 提取的实体列表、置信度 |
| RAG 召回 | 召回的 citations 列表（法条ID/案例ID/规则ID） |
| 模型生成 | 原始输出（含未校验的 citations）、语义校验评分 |
| 规则校验 | 引用校验结果、禁用词命中、删除结论列表 |
| 最终输出 | 最终呈现给用户的结论及对应 citations |
| 版本快照 | 法规库版本、案例库版本、规则库版本、模型版本 |

### 4.2 版本管理

**知识库版本化**：每次法规更新、案例库扩充、规则库修改，均生成新版本快照并记录版本号。用户的每一次分析结果绑定当时的版本快照，确保可复核、可对照、可追溯。

**模型版本锁定**：固定 AI 模型版本，不做静默热更新。每次模型更换需在测试环境通过全套回归测试后方可上线。

### 4.3 审计与合规自证

当出现用户投诉或外部审查时，系统可提供：
- 该次分析的完整 Trace 日志
- 所有 citations 的原始数据来源证明
- 规则引擎的判定路径记录（为何触发了律师介入提示）
- 知识库版本证明（用的是哪个时间点的法规库）

## 五、关键技术选型建议

| 技术组件 | 推荐选型 | 选型理由 |
|---------|---------|---------|
| 语言模型 | ChatGLM3 / 通义千问（闭域部署） | 支持私有化部署，上下文可控 |
| 向量检索 | Milvus 或 FAISS | 开源成熟，支持千万级向量 |
| 规则引擎 | Drools 或 JsonLogic | 规则可版本化，便于法律专家编写 |
| 语义校验模型 | 轻量级 BERT 微调分类器 | 检测隐含判断，计算开销低 |
| 数据库 | PostgreSQL + Elasticsearch | 结构化+全文检索双引擎 |
| 前端框架 | React/Vue | 成熟生态，组件化 |
| 后端框架 | FastAPI | 轻量异步，适合 AI 场景 |

## 六、架构决策总结

法眼 AI 的架构核心不是"如何让 AI 更聪明"，而是**"如何让 AI 在边界内工作"**。整个架构的设计逻辑可以归结为三条主线：

**第一条：AI 的能力边界由规则引擎划定，而非模型自身。** 复杂度判定、律师介入触发、降级兜底——所有关键判断均通过硬编码规则实现，模型无权自行决定。

**第二条：所有输出的可信度由引用机制保证，而非由模型保证。** 每一条结论必须有 citations，每一条 citations 必须经过存在性校验，无引用则不输出。

**第三条：系统的可信度由审计机制背书，而非由承诺背书。** 完整的 Trace 日志、版本快照、规则路径记录，使系统在面对投诉和审查时可自证清白。

这三条主线的共同指向是：**法眼不是一个"聪明的"法律 AI，而是一个"可信赖的"法律 AI 系统。** 在法律场景中，后者比前者更有价值，也更难被复制。
